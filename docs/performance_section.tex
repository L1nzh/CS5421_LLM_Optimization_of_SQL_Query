\section{Performance Evaluation}
\label{sec:performance}

\subsection{Experimental Set-up}

We evaluate the proposed LLM-based SQL optimization pipeline on the TPC-DS decision-support benchmark using a PostgreSQL back-end populated with the SF=1 dataset. The experimental workload is taken from the 99 benchmark queries stored in the testing set, and each query is executed against the populated PostgreSQL instance to provide both semantic validation and physical performance measurements. Since the goal of the system is not merely to generate alternative SQL text, but to generate semantically equivalent rewrites that improve execution behavior, our evaluation combines correctness checks with runtime-oriented and resource-oriented metrics.

The evaluation is organized into two phases. In Phase~1, we perform a controlled screening of prompt and reasoning combinations in order to identify the most promising configuration. Ten queries are randomly sampled from the 99-query pool using a fixed seed, and each query is evaluated under all available prompt/reasoning combinations for the selected model. In the collected result set, this produced 12 combinations corresponding to the Cartesian product of four prompt strategies ($P0$--$P3$) and three reasoning strategies ($R0$--$R2$), all instantiated with \texttt{gpt-5-mini}. In Phase~2, the best-performing configuration selected in Phase~1 is applied to a larger random sample of 30 TPC-DS queries. In both phases, the system generates three candidate rewrites per query and benchmarks each query configuration with five execution trials so that timing statistics can be aggregated robustly.

The performance evaluation uses the following metrics. First, semantic validity is enforced through result-set equivalence checking, which acts as a hard gate before any speed claim is accepted. Second, execution performance is measured by the median execution time reported by \texttt{EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)}. Third, candidate speedup is defined as the ratio between the baseline execution time and the candidate execution time; values larger than 1.0 indicate improvement. Fourth, resource efficiency is captured through PostgreSQL buffer statistics, which are converted into a derived memory score based on cache-hit behavior and temporary spill avoidance. Finally, ranking uses a composite score that combines speedup and memory score with weights $0.7$ and $0.3$, respectively. Because the collected results contain several large outliers, we report medians alongside means and interpret medians as the primary summary statistic.

\subsection{Phase~1: Comparative Screening of Prompt/Reasoning Combinations}

Phase~1 is intended to answer a model-selection question: among the available prompt/reasoning combinations, which configuration provides the strongest balance between correctness and performance? The macro-level comparison is shown in Figure~\ref{fig:phase1-leaderboard}, while Figure~\ref{fig:phase1-heatmap} summarizes the interaction between prompt strategy and reasoning strategy.

At the combination level, the best configuration is \texttt{P3\_\_R0\_\_gpt-5-mini}. This combination achieves a valid selection rate of 90\% and the highest median selected speedup among all screened combinations, at 1.267$\times$. The second-best configuration is \texttt{P2\_\_R1\_\_gpt-5-mini}, which also maintains a 90\% valid selection rate but with a slightly lower median speedup of 1.213$\times$. By contrast, weaker combinations such as \texttt{P1\_\_R0\_\_gpt-5-mini} and \texttt{P1\_\_R1\_\_gpt-5-mini} produce near-parity median speedups (0.996$\times$ and 0.982$\times$, respectively), indicating that engine-aware prompting alone is not sufficient to drive consistent gains.

At the prompt-strategy level, $P3$ delivers the highest median speedup (1.250$\times$), followed by $P2$ (1.114$\times$). This suggests that richer schema and statistics context helps the model discover more aggressive and more beneficial rewrites. However, the gain does not come without a trade-off: the overall valid selection rate of $P3$ drops to 76.7\%, while $P2$ retains a better balance at 86.7\%. Thus, $P3$ appears to increase optimization power but also increases the risk of invalid or unstable rewrites when paired with less suitable reasoning modes.

At the reasoning-strategy level, $R2$ (two-pass prompting) produces the highest median speedup (1.117$\times$), but its valid selection rate is only 65\%, substantially lower than the 90\% achieved by both $R0$ and $R1$. This result indicates that explicit multi-step reasoning can improve aggressiveness, but it also increases the chance of generating semantically unsafe or operationally brittle rewrites. In contrast, $R0$ and $R1$ are markedly more reliable, with $R0$ ultimately forming the best overall combination when paired with $P3$.

Taken together, these results motivate the selection of \texttt{P3\_\_R0\_\_gpt-5-mini} for Phase~2. It preserves the high validity of direct prompting while still benefiting from the richer structural information supplied by the $P3$ schema/statistics prompt.

\subsection{Phase~2: Evaluation of the Best Configuration}

Phase~2 applies the selected configuration, \texttt{P3\_\_R0\_\_gpt-5-mini}, to a larger set of 30 randomly sampled TPC-DS queries. The aggregate result is more mixed than the Phase~1 screening might suggest. Although the system always produces a ranked selection, only 24 of the 30 selected candidates are semantically valid, yielding a valid selection rate of 80\%. Across the valid candidates, the median speedup is only 1.002$\times$, indicating that the typical case is near parity with the baseline rather than a large improvement. The mean speedup is much larger at 5.359$\times$, but this average is heavily inflated by a few extreme wins and therefore is not representative of the central tendency. This gap between mean and median confirms that the performance distribution is heavy-tailed.

The outcome distribution shown in Figure~\ref{fig:phase2-outcomes} makes this pattern explicit. Among the 30 evaluated queries, 5 are strong improvements (speedup $\geq 1.10$), 9 are mild improvements, 3 are near parity, 6 regress, 1 valid candidate lacks a computable speedup due to missing baseline benchmark timing, and 6 selected candidates are invalid. Therefore, while the best Phase~1 combination generalizes reasonably well in terms of producing valid candidates, it does not produce universal runtime gains. Instead, it exhibits selective effectiveness: substantial wins on some queries, modest or neutral impact on many, and clear regressions on a non-trivial subset.

\subsection{Analysis of Why the Best Combination Succeeds or Fails}

The micro-level results reveal that the best configuration is highly effective on a small number of queries but not uniformly effective across the workload. The strongest improvement occurs on \texttt{query\_81}, which achieves a speedup of 94.66$\times$, followed by \texttt{query\_31} at 7.47$\times$, \texttt{query\_69} at 1.52$\times$, \texttt{query\_54} at 1.32$\times$, and \texttt{query\_83} at 1.28$\times$. These large wins explain the inflated mean speedup reported earlier. At the other end of the spectrum, the worst regression occurs on \texttt{query\_72} (0.493$\times$), followed by \texttt{query\_51} (0.750$\times$) and \texttt{query\_71} (0.783$\times$). This spread suggests that the chosen prompting strategy can discover highly beneficial rewrites when the original query contains obvious optimization opportunities, but that the same strategy can also over-structure or over-constrain queries that were already close to a good physical form.

The structural-change analysis provides a plausible explanation for this behavior. Figure~\ref{fig:phase2-structural} compares the rewrite patterns associated with strong improvements and regressions. Strong improvements are most commonly associated with introducing \texttt{MATERIALIZED} CTEs and reducing subquery count. This is consistent with the intuition that some TPC-DS queries benefit when repeated sub-expressions are explicitly materialized or when nested expressions are simplified. By contrast, regressions are most often associated with adding more CTE structure and introducing extra CTE layers without corresponding simplification. This suggests that gratuitous decomposition---even when semantically valid---can lead PostgreSQL to choose a less efficient plan.

This finding is also visible in the speedup-memory relationship shown in Figure~\ref{fig:phase2-speedup-memory}. Some of the strongest wins preserve or improve memory behavior, but not all structurally complex rewrites do so. In particular, several regressions combine modest or poor speedup with only moderate memory scores, indicating that additional query structure does not automatically translate into better resource efficiency. Similarly, Figure~\ref{fig:phase2-runtime-buckets} suggests that larger baseline runtime does not guarantee larger benefit: while some expensive baseline queries admit dramatic improvements, others remain difficult to optimize and may even regress.

Overall, these results suggest that the selected best combination is \emph{conditionally effective}. It performs well when the model identifies a beneficial structural simplification, targeted materialization opportunity, or redundant subquery pattern. It performs less well when it introduces additional query scaffolding without sufficiently improving the execution plan. This explains why the same configuration can produce both substantial improvements and meaningful regressions across the Phase~2 workload.

\subsection{Threats to Interpretation and Data Quality}

Finally, the analysis also exposes several evaluation caveats that should be stated explicitly. First, a small number of results have missing speedup values because the baseline benchmark timing is absent even though a selected candidate exists. In the collected Phase~2 data this affects one selected query directly, and in the broader trace-level summary there are five selected rows with missing speedup information. Second, six selected Phase~2 candidates are invalid, which highlights that even the best screened combination still requires a strict semantic validation gate. Third, because the speedup distribution is extremely skewed, arithmetic means can overstate the apparent average improvement. For this reason, medians and outcome buckets provide a more reliable description of typical behavior.

In summary, the empirical evaluation shows that the proposed pipeline can indeed produce meaningful performance gains on a subset of TPC-DS queries, and that richer prompt context (\texttt{P3}) coupled with direct generation (\texttt{R0}) is the strongest configuration among those tested. At the same time, the results also show that optimization benefits are not uniform across the workload, that some valid rewrites still regress, and that the best practical system design must combine LLM generation with strict validation and careful performance-based selection.
